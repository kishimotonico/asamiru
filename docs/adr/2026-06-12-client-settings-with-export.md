# ADR: 設定はクライアント保持のまま、バックアップはエクスポート/インポートで行う

**日付**: 2026-06-12  
**ステータス**: 採用（サーバー設定永続化の revert を含む）

## 背景

2026-06-11 に「設定をサーバー側 JSON ファイルに永続化し、localStorage をキャッシュに格下げする」実装を行った（`adfe4c8`、計画は [server-settings-plan](../issues/2026-06-11-server-settings-plan.md)）。kiosk のブラウザプロファイル消失で設定が飛ぶリスクへの対応だった。

オーナーレビューで次の判断により revert した（`adfe4c8` の revert コミット参照）。

- **複数デバイスで違う設定を表示したい需要がある**。サーバー権威にすると全クライアントが同一設定に収斂してしまい、これはバグではなく仕様の損失になる
- **サーバーの役割を増やしたくない**。API は「鉄道情報の取得・正規化」「モニター制御」という外部境界の仕事に限定し、UI の関心事を持ち込まない
- 起動時の `GET /api/settings` 依存が増え、起動経路が複雑化する

## 決定

1. 設定は従来どおり各クライアントの localStorage（jotai `atomWithStorage` + `mergedStorage`）が唯一の保存先とする
2. 「設定が消えやすい」問題には、**設定の JSON ファイルへのエクスポート / インポート**機能で対応する（設定モーダルから手動操作）
3. サーバーに設定関連のルート・ストレージは置かない

## トレードオフ

- バックアップは自動ではなく手動になる。設定を大きく変えたときにエクスポートし忘れると消失リスクは残る（個人用途として許容。設定項目は再入力可能で、最も面倒な ICS URL もカレンダーサービス側で再取得できる）
- デバイス間で設定を揃えたい場合はエクスポート→インポートの手動同期になる（これは「揃えない自由」の裏返しで意図どおり）

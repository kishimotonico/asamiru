# 発車情報の表示本数と候補探索上限の修正

## 背景

交通カードの発車情報で、表示本数を `3` にすると本来表示できるはずの列車が `2` 本しか出ず、`4` にすると表示されるケースがあった。

## 原因

`apps/api/src/departures.ts` は、`displayCount` をもとに方向ごとの `dia` 確認上限を決めている。

従来の式は `min(10, max(displayCount + 4, displayCount * 2))` で、`displayCount = 3` の場合は最大 `7` 件、`displayCount = 4` の場合は最大 `8` 件だけ候補を確認していた。

乗車駅に停車しない列車、発車済み列車、分岐先違いの列車が候補に混ざると、`3` 本指定時だけ探索が足りずに `2` 本で打ち切られることがある。

## 対応

方向ごとの候補確認数を `MAX_DIA_CHECKS_PER_DIRECTION = 10` の固定上限にした。

これにより、表示本数が少ない場合でも最大 `10` 件までは候補を確認し、表示本数を増やしたときだけ探索が深くなる挙動をなくす。

上流負荷対策として、`dia/{trainId}.json` の取得にも in-flight 共有を追加した。
同じ `serviceDate + trainId` の時刻表を同時に必要とした場合、後続リクエストは先行 Promise を await し、上流への重複リクエストを出さない。

## 検証

- `pnpm build`
- 開発 API で `京王多摩センター` / `displayCount = 3` と `4` を確認し、下り方面がそれぞれ `3` 本 / `4` 本で返ることを確認
- 開発 API で `明大前`・`調布` / `displayCount = 3` と `4` を確認し、上下方面がそれぞれ指定本数で返ることを確認

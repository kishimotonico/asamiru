# 駅別時刻表補完実装レビュー

対象コミット: `28c2721 feat: 駅別時刻表データによる発車案内の補完機能を追加`

## 指摘

### 1. scraper が取得失敗を握りつぶして不完全な `timetable.json` を生成する

`apps/api/scripts/scrape-timetable.ts` の `fetchNavitime()` は HTTP エラーや fetch 例外を `console.warn("[skip] ...")` して `null` を返し、呼び出し側も `continue` する。
これは「年1回生成してコミットする静的マスタ」としては危険で、API 側の不調やリンク抽出ミスを欠落データとして固定化してしまう。

スクレイピングは手動バッチなので、失敗時は fail-fast で終了するべき。
少なくとも対象駅・方向のカバレッジ検証を最後に行い、期待駅の時刻表が欠けていれば非ゼロ終了にする。

### 2. 生成データの駅名が設定駅名と一致せず、補完されない駅がある

実データでは以下が確認できた。

- `KEIO_STATIONS` には `飛田給` があるが、`timetable.json` は `飛田給（味の素スタジアム前）`
- `KEIO_STATIONS` には `新線新宿` があるが、`timetable.json` に存在しない
- `timetable.json` にはアプリ対象外の `多摩動物公園` / `府中競馬正門前（東京競馬場前）` が含まれる

`buildScheduleCandidates()` の `normalizeKey()` は NFKC と `ヶ/ケ` だけなので、`飛田給` は一致せず補完されない。
ユーザーが選択可能な駅と生成時刻表の駅キーを同じマスタから検証する仕組みにすること。

### 3. 行先注記の除去対象が実データと違う

`normalizeDestination()` は `【...】` だけを除去するが、NAVITIME の実データは `京王八王子〔高幡不動から各駅停車〕` のように `〔...〕` を使っている。
そのため生成済み `timetable.json` に注記付き行先が大量に残っており、UI で `橋本〔京王多摩センターから各駅停車〕行` のように表示されうる。

重複排除は `trainNo` 主体なので即二重表示にはなりにくいが、表示品質と将来のフォールバックキーの保守性を落とす。
`〔...〕` / `【...】` の両方を除去し、生成済み JSON も再生成する。

### 4. 補完ロジックのテストがない

`pnpm build` は通るが、今回の主なリスクは型ではなく日付・駅名・行先正規化・重複排除。
最低限 `timetable.ts` の純粋関数をテスト可能にし、以下を固定テストにする。

- `飛田給` と `飛田給（味の素スタジアム前）` の一致
- `〔...〕` 注記の除去
- `00:xx` を運行日 24時台として比較すること
- `trainNo` と realtime `trainId.trim()` の重複排除

## 確認済み

- `pnpm build` は成功。
- `apps/api/dist/data/timetable.json` は生成されており、build 後の JSON 同梱自体は問題なし。
- 補完挿入位置は `groupDepartures()` 前で、前回指摘した `trainId` / `scheduledMinutes` 喪失問題は回避されている。
